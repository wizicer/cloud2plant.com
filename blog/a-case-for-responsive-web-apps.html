A case for responsive web apps

July 24, 2014

The mobile revolution

Have you ever noticed all those people staring at their phones? In public, in the car, in line at the store? They never seem to take their eyes off their phone. What do you think they’re looking at?

Whether you like it or not, the mobile revolution is here. You spend so much time, effort, and money trying to get visitors to your website, why not make their experience a pleasant one? (Hint: viewing a website shrunk down to 10% on a phone isn’t a pleasant experience.)

By now, you’ve likely heard of responsive web design. If not, it essentially means that your website will “respond” to your visitors’ screen size. Here’s an example of our app on a desktop and the same page on a phone:

Through the use of CSS media queries, the website is built to adapt to virtually any screen size. This can drastically improve your visitors’ experience—especially the ones glued to their phones.

This method has been applied to tons and tons of websites already. But what about web apps?

Responsive web apps

It’s disappointing to load a web app on your phone when it was built for a desktop. To be frank, the experience just plain sucks. Some developers argue that building a separate mobile app is too time consuming, usually limited in functionality, or just plain silly.

But you don’t need to build a separate app anymore, and we have proof.

When we designed the latest version of our web app, we started from the ground up with the difficult requirement of making it fully responsive. Every page, every feature—everything. If it worked on a desktop, it should work equally as well on a tablet or phone. Whether the user could only click or tap, it shouldn’t prevent them from using the app and it shouldn’t make for a bad experience.

That turned out to be quite a challenge, but nothing we couldn’t overcome with a bit of perseverance. Granted, the world is still in that awkward native app vs. web app phase. Native apps are sexy. They’re faster. They’re better. Right?

Sure, but the HTML5 movement is pushing forward fast. There are apps that let you develop in HTML, CSS, and JavaScript and let you publish a native app—many offer access to device-specific APIs so you can access cameras, filesystems, etc. Firefox OS is a mobile platform that uses HTML, CSS, and JavaScript for its apps. Chrome browser extensions use the same technologies.

HTML isn’t going away—it’s just getting started.

That remark might make you think of when Facebook pulled the plug on HTML5 in favor of native apps, but that was two years ago. Things are getting better. Devices are getting faster. More JavaScript APIs are becoming available. Two years ago, we couldn’t have built the app we built today either.

The fascinating thing about Facebook is that they had a functional app based on HTML5 that millions of people used. There were some gripes, but things worked pretty well otherwise. If they hadn’t jumped ship, things would have gotten significantly better.

The native app dilemma

Without turning this into a native app vs. web app argument, let’s look at a few important points that you’ll want to consider if you’re looking to build a native app.

Which platforms are you going to target? What devices will be supported? What languages will you need to know to develop your app for those platforms and devices? Does your company have developers with experience in all those languages?

As a bootstrapped company, this is why we chose to build a responsive web app. There was no way we could bring on developers for every device our users would want us to support. By building a web app, we were able to use technologies that we already understood. We were able to build a single version of the app that worked on any device, regardless of platform.

If we tried to build a native app, we’d most likely still be working on it. (And that would be just the first one!)

Efficiency rules

The resources we saved by building a responsive web app allowed us to accommodate desktops, tablets, and phones much faster than the native approach. We didn’t have to worry about hiring new people and bringing them up to speed. We didn’t have to buy and learn new hardware and software. We didn’t have to design a multitude of different apps (nor keep them all updated as new features get rolled out). It wasn’t exactly easy, but it was still easier and more efficient than the alternative.

In hindsight, we feel that we made a great decision to stick with a web app. We started with a responsive framework that was reliable and actively developed. We spent a lot of time tweaking and polishing the interface so it worked great on virtually any screen. We tested things thoroughly on a number of emulators and physical devices. The result was an app that “just works” on whatever browser you happen to be using. We were even able to use our app on a Nintendo Wii!

It may not be for everyone, but before you convince yourself that a native app is the only way to go, reconsider what you could achieve with a web app instead. You might be surprised at what you can do with HTML, CSS, and JavaScript these days.

Maybe you’re wondering how our responsive web app would compare to a native version? In all honesty, It may not be quite as good—but it would definitely give it a run for its money.
